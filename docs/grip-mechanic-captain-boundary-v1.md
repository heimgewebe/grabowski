# Grip Mechanic/Captain Boundary v1

Status: implementation slice
Date: 2026-07-07 (updated 2026-07-14: atomic merge lease guard)

## Boundary

`mechanic-loop` is the normal-action role. It may dispatch only known normal grips, and every action must carry a target object, scope object and relative receipt path. Unknown actions, recursive dispatch, Captain-only grips, missing target, missing scope, missing receipt path, target/parameter mismatch and high-impact actions fail closed. If any child action is blocked, the parent `mechanic-loop` receipt is blocked as well; `continue_on_blocked` only controls sequencing, not success semantics.

`captain-preflight` is a Captain-only seam. It recognizes high-impact action envelopes, validates target, scope, recovery or irreversibility, and target-change records. It remains read-only: it evaluates the Captain authority gates listed below and never executes anything.

`captain-run` is the execution seam for action-specific Captain operations. It executes exactly one action per invocation and currently supports `pr-merge` plus the registered `grabowski-self` adapter for `runtime-deploy`. Both paths require the same Captain gates, `allow_execution=true`, a fresh action-bound `execution_intent` (see "Captain execution intent" below), and evidence bound to the reviewed `expected_head`. The merge executor verifies GitHub state before and after mutation. The self-deploy executor verifies the canonical Grabowski checkout before scheduling an independent delayed deployment job and verifies the scheduling receipt; deployment completion remains a later, separately observed state. In trusted-owner mode, a reversible `runtime-deploy` may execute without a per-action human authorization record when every Captain evidence gate passes, the exact trusted-owner autonomy policy is present, and the action remains bound to the registered local adapter and target. This is intentional operator authority, not an implicit exemption. Other high-impact actions remain explicitly blocked until they receive dedicated executors.

## Captain authority gates

`captain-preflight` and `captain-run` evaluate a structured gate list per request: `high-impact-marked`, `target-bound`, `scope-bound`, `target-change-record`, `recovery-or-irreversibility`, `status-projection-fresh`, `execution-authority-present`, `review-evidence-present`, `diff-bound`, `ci-green`, `autonomy-policy`, `human-authorization-present`. Every gate reports `id`, `status` (`pass`/`blocked`), `reason` and optional `details`. Missing safety evidence blocks; the evaluation is fail-closed.

Target binding is action-specific: `pr-merge` requires exactly one syntactically bounded `owner/repo` slug, a positive integer `pr`, and a concrete non-wildcard `base`; `runtime-deploy` requires exactly one concrete `repo` or `service`, exactly one concrete `environment` or `runtime_target`, and one registered `adapter`; the first executable adapter is `grabowski-self`, bound to `grabowski-mcp`/`heimgewebe/grabowski` on `heim-pc`; `service-restart` requires one concrete non-wildcard `host` and one concrete non-wildcard `unit`; `fleet-mutation` requires a concrete `fleet_target` and an explicit non-generic `operation`; `cleanup-apply` requires one concrete `cleanup_target` plus a bounded `repo` or concrete `checkout_path`, and any provided optional target field must be typed and concrete. Target changes require an explicit non-empty `target_change` record when present or required; silent target switches and empty target-change records are rejected.

Status projection is mandatory evidence: `status_projection` must be a non-empty object with `status_projection_fresh=true`, an allowlisted `status_projection_source` label that matches the hash-bound `status_projection.source`, a supported `schema_version`, required minimal typed status fields including `healthy=true`, a timezone-aware ISO-8601 string or numeric Unix-timestamp `generated_at` inside the configured max-age window with bounded clock-skew tolerance, replay-reference metadata such as `receipt_ref`, `run_id` or `nonce`, and a `status_projection_sha256` that matches the deterministic JSON hash of the projection object. A missing, stale, non-allowlisted, unhashed, replay-reference-less, unhealthy or drifted projection blocks. Caller-declared freshness is not enough by itself. Replay-reference metadata is presence-checked but not uniqueness-verified; when multiple replay-reference fields are present, `receipt_ref` is preferred over `run_id`, and `run_id` is preferred over `nonce`. `source_trusted` is retained for compatibility; new callers should use `source_allowlisted`. Captain evidence must be bound to an explicit `expected_head`; review and CI evidence must match that head. The projection is evidence about observed state, never runtime truth.

Decision semantics are split by grip. `captain-preflight` always keeps top-level `decision`, `status`, and `receipt_status` as `blocked` because it is observational. When all gates pass in manual mode, `gate_decision` becomes `ready_for_manual_captain_decision`. When `trusted_owner_mode=true` and `autonomy_policy=act_unless_irreversible_or_ambiguous` are present for actions that explicitly declare `risk.irreversibility=reversible` and have an implemented Captain executor, `gate_decision` becomes `ready_for_autonomous_captain_execution`; per-action `execution_authority` and `human_authorization` evidence are then not required. Missing or ambiguous irreversibility records and irreversible actions are never covered by trusted-owner autonomy and still require explicit authorization evidence.

`captain-run` may return `decision=executed`, `status=passed`, and `receipt_status=passed` only for synchronous implemented executors whose post-action verification passes. A verified delayed self-deployment returns `decision=scheduled`, not `executed`: this proves only that the independent job was registered with the expected head and observation handles. Scheduling is serialized by an owner-controlled lock with bounded acquisition time; a non-terminal job with the same runner, repository and expected head is reused even when the requested delay differs instead of registering a duplicate. A running self-deploy for another head, any non-terminal self-deploy with an uncertain state, multiple identical running jobs, malformed job identity or receipt paths, and uncertain matching outcomes block fail-closed. Terminal completed, succeeded, failed and launch-failed jobs do not permanently prevent an explicit retry. The current executable action allowlist is deliberately narrow: `pr-merge` and `runtime-deploy`; the latter has only the `grabowski-self` adapter. Its receipt distinguishes a newly observed local job registration (`local_mutation_observed=true`) from deployment completion (`remote_mutation_observed=false` until later observation), and preserves an unknown local mutation outcome when scheduling raises after invocation. `captain-run` v1 rejects multiple actions because review, diff, CI and status-projection evidence are still top-level and not per action. The PR merge executor first observes the PR with `gh pr view`, requires all required PR fields to be present, allows a bounded preflight settle loop for transient view failures or `UNKNOWN` mergeability / merge-state values, then requires `state=OPEN`, `isDraft=false`, matching `headRefOid`, matching `baseRefName`, `mergeable=MERGEABLE`, and `mergeStateStatus=CLEAN`. The settle loop only retries transient observation or UNKNOWN-state errors; hard blockers such as mismatched head, mismatched base, draft state, non-open state, already-merged state, or missing required fields stop before any merge command. It then reads the bound repository's current GitHub merge policy, validates all three method flags, selects the first allowed method in the deterministic order `merge`, `squash`, `rebase`, and invokes `gh pr merge <selected-method> --match-head-commit <expected_head>` against the bound `owner/repo`. Missing, malformed or fully disabled policy blocks before mutation. Finally it verifies `state=MERGED`, the same `headRefOid`, the same base, and a valid `mergeCommit.oid` using a bounded post-merge observation loop. A PR that is already merged before execution is blocked in this slice instead of being treated as already satisfied. If the merge command fails, raises through the runner, succeeds unclearly, or post-merge verification fails after bounded retries, `captain-run` returns a failed receipt that still preserves the invocation record, command-return state, bounded command output digests/previews, and emits `execution-preflight`, `execution-attempted`, and `post-execution-verification` checks. A nonzero merge command is always a primary failure even if a later observation sees the PR as merged. Pre-execution drift returns a blocked receipt with no merge attempt. `allow_execution=true` alone still never releases execution; it is only one execution precondition after the evidence gates pass.

The execution receipt separates invocation, command return and verified mutation. `execution_invoked=true` means the bounded merge command runner was called. `command_returned=true` means the runner returned a command result object. `execution_attempted=true` means the merge command returned and was therefore observable as a command attempt. `remote_mutation_observed=true` means post-execution observation verified the PR as merged; it does not prove that the merge command returned successfully. Top-level `execution_counts` reports `invoked_count`, `command_returned_count`, `attempted_count`, and `verified_count` separately. `preflight_view_summary` records bounded pre-merge observation attempts, and `verify_view_summary` records bounded post-merge verification attempts.

### Atomic merge lease guard

Immediately before the irreversible `gh pr merge` dispatch, the Captain path performs a second uncached PR observation and reads both the complete GitHub diff and the PR file inventory. The observation must still show the exact PR number, open non-draft state, expected head, expected base branch, a valid base SHA, `MERGEABLE`, and `CLEAN`. The complete non-empty patch is canonicalized as UTF-8 with exactly one terminal newline and must hash to the already reviewed `diff_sha256`. The reported `changedFiles` count must equal the returned file inventory, the inventory must stay within the explicit 128-path and encoded-size bounds, every path must be a canonical relative repository path, and every entry must have an allowed `ADDED`, `MODIFIED`, or `DELETED` change type. Rename and copy entries block fail-closed because the compact observation does not prove both the old and new path. Caller-supplied lease snapshots, path inventories, or earlier guard receipts are rejected; they cannot be replayed as current evidence.

The guard resolves the canonical local repository through Git's shared common directory so linked worktrees bind to the same repository identity. It derives seven effect resources: a GitHub-repository component, the base- and head-branch components, repository-main service, exact PR service, base-branch merge gate, and GitHub deployment target. The sorted changed-path inventory is separately bound by SHA-256 and resolved beneath the canonical repository root. `execution_intent.context.lease_owner_id` remains part of the reviewed, hash-bound intent, but the MCP production surface no longer trusts it as an owner credential. Immediately before dispatching `captain-run`, the server derives a stable owner identifier from the live MCP session and effective profile, signs a five-minute actor assertion with a process-local secret, and injects that assertion through a reserved internal parameter. The merge guard verifies the assertion and replaces the caller-declared owner with the authenticated runtime actor. A caller-supplied assertion, a missing MCP context, an expired or altered assertion, or an invalid actor identity blocks before dispatch. Direct Core calls without this server assertion retain the legacy declaration only for isolated tests and explicitly keep `server_authenticated_lease_owner_identity` in `does_not_establish`.

Inside one `BEGIN IMMEDIATE` resource-database transaction, Grabowski reads all live relevant repository, changed-path, exact-scope, branch, service, deployment and merge-gate leases and records their public snapshots. A broad repository lease, an overlapping file or directory lease, a mutating same-repository scope without exact paths, or an already active merge effect blocks. A foreign lease on a provably disjoint path in the same repository remains allowed and is not absorbed into the guard snapshot. On the MCP path, only relevant leases owned by the authenticated runtime actor receive the same-owner exemption; copying another visible owner into the execution intent no longer changes that decision. Relevant leases under any other owner, including a caller-declared task owner, remain foreign and block. The guard creates only its own missing critical effect resources. While the merge gate is live, later broad or path-overlapping acquisitions are rejected regardless of owner, while exact disjoint path work may continue; a second merge on the same base remains serialized by the gate. The guard remains held through the existing post-merge GitHub readback and releases only the resource keys it created; pre-existing leases are never released, expired, reconciled or force-overridden.

The `merge_lease_guard` receipt binds repository slug, PR number, base branch and base SHA, head SHA, complete diff SHA-256, sorted changed paths and their SHA-256, execution-intent SHA-256, canonical local repository, ordered resource-key digest, observed and newly acquired lease snapshots, snapshot digest, nanosecond observation, dispatch and completion timestamps, and a bounded `lease_owner_binding` record. That record exposes only the binding source, an authentication boolean, and the SHA-256 of the verified assertion; it never returns the raw proof. A merge observed externally after this actor was blocked is recorded as `external_merge_observed=true`, while `contract_satisfied=false` and `dispatch_called=false` remain unchanged. This prevents duplicate dispatch and does not manufacture retrospective compliance.

The lease guard is necessary but insufficient for merge. It does not replace authority, current user-visible diff delivery, critical review, green CI, branch protection, recovery evidence, `--match-head-commit`, or post-merge verification. It coordinates cooperating Grabowski actors; it cannot lock a non-cooperating external GitHub actor.

Recovery and irreversibility records are preconditions for any later mutation. Reversible actions require a non-empty `recovery_path`; irreversible actions require a non-empty `irreversibility_record`. These records do not prove that a later execution would be safe. Scope must contain a concrete effect or boundary record; `max_targets` alone is only a quantity limit and is not a sufficient scope boundary.

## Captain execution intent

`captain-run` additionally requires one structured `execution_intent` parameter per invocation, validated before any executor is called; `captain-preflight` does not require it because it never executes. The intent is a canonical object with exactly the fields `schema_version` (`1`), `kind` (`grabowski_captain_execution_intent`), `action`, `target_sha256`, `expected_head`, `expected_base`, `evidence_sha256`, `actor`, `context`, and `issued_at`; unknown top-level fields are rejected. It must bind the single requested action by name, the canonical action target digest (`target_sha256`, the deterministic JSON SHA-256 of the bound target), the lowercase 40-hex `expected_head`, and the action-specific `expected_base`: the PR base branch for `pr-merge`, the bound `environment`/`runtime_target` value for `runtime-deploy`. `evidence_sha256` must contain exactly the decisive evidence digests plus a deterministic `authorization_sha256` over execution authority, human authorization and trusted-owner policy state `actions_sha256`, `status_projection_sha256`, `diff_sha256`, `review_evidence_sha256`, and `ci_evidence_sha256`; the projection, review and CI digests are recomputed as deterministic JSON SHA-256 over the supplied evidence objects and compared against the declared values. `actor` must be a non-empty object with a non-empty string `id`; `context` must be a non-empty object. `issued_at` must be a timezone-aware ISO-8601 timestamp; it is normalized to UTC and plausibility-checked against a 600-second maximum age and a 120-second future clock-skew tolerance. All hex fields must already be lowercase; uppercase hex is rejected as non-canonical.

A missing, malformed, non-canonical, stale, future-dated, unverifiable, or drifting intent — drift against the bound action name, target digest, expected head, expected base, or any decisive evidence digest — blocks fail-closed before any GitHub or deployment runner is invoked; the blocked receipt keeps `executions` empty. When more than one action is requested, the intent is rejected with `execution_intent_requires_exactly_one_action` in addition to the existing single-action boundary. The intent check is one more release condition next to the unchanged Captain gates, `allow_execution`, the executable allowlist and the exactly-one-action boundary; it never replaces or weakens them, and a valid intent alone never grants execution.

Receipts never echo the raw intent, actor or context. The top-level `execution_intent` receipt record carries only fixed error codes, freshness limits, and normalized non-sensitive values: the action name, expected head, expected base, target digest and evidence digests only after they verified equal to the bound request, the normalized UTC `issued_at` with its observed age, plus digests `intent_sha256` (deterministic JSON SHA-256 of the presented intent), `actor_sha256` and `context_sha256`. Each per-action `captain_receipt` binds `execution_intent_sha256` whenever an intent was presented, and the `execution-intent-bound` receipt check reports pass or fail with codes and digests only. A valid execution intent proves operator-declared binding at issuance time, not execution safety.

## Action evidence schemas

Each Captain action record carries an `evidence_schema` with schema version, concrete target binding, required evidence objects and digest-binding fields. The schema for `pr-merge` binds repo, PR, base, expected head, diff, CI, review and authorization evidence. Runtime deploy, service restart, fleet mutation and cleanup schemas bind their action-specific targets to projection, rollback/recovery, restart-budget or dry-run evidence as appropriate. See `docs/captain-action-evidence-schemas-v1.md`. The schema is a contract and non-claim, not proof of safety or execution authority.

## Does not establish

This slice does not establish automatic deploy completion, service restart safety, fleet mutation safety, cleanup safety, runtime correctness, semantic correctness, review completeness, production safety, or privileged execution outside the bound `captain-run` receipt. A valid `execution_intent` establishes only that the operator bound this exact action, target, head, base and evidence at a plausible recent time; it does not establish execution safety and never releases execution by itself. Scheduling a Grabowski self-deployment is not proof that the delayed job completed; job status, logs, and the new runtime identity must be checked after reconnect. Further Captain follow-ups (additional deploy adapters, restart/fleet/cleanup executors, webhook or dispatcher integration) remain separate tasks.
